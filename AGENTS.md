# AGENTS.md

本文件定义 OrnnSkills 仓库内 AI Agent 的默认产品方向与执行规则。除非用户明确要求偏离，否则后续设计、文案、实现、评审都应遵守以下原则。

## 1. 产品方向

### 1.1 V2.0 定位

OrnnSkills V2.0 的默认定位是:

**开源、本地优先、跨宿主、可视化的 Skill 生命周期与演进管理器。**

重点是 `skills are first-class citizens`，不是通用 AI 平台，也不是 prompt / MCP / plugin 的全能管理器。

### 1.2 核心价值排序

V2.0 的价值排序如下:

1. 演进管理
2. 使用透明度与价值认知
3. 生命周期管理
4. 可视化与易用性

取舍原则:

- 如果一个功能不能明显增强 skill 的可见性、可管理性、可演进性，应谨慎进入主路径
- 如果一个功能会把产品推向团队协作平台、云端治理平台或通用 AI 客户端，应默认拒绝

### 1.3 默认用户

- 个人开发者
- 本地优先用户
- 多宿主重度用户

默认不为团队协作、审批流、权限系统、云端工作台做产品设计。

## 2. 产品边界

### 2.1 V2.0 P0 能力

以下能力默认视为 V2.0 P0:

- 扫描所有项目、所有宿主的 skills
- 统一索引、展示、搜索、筛选
- 调用频率、活跃度、覆盖范围、价值信号
- 评估调用效果与演进机会
- 版本、diff、快照、回滚
- 安装、卸载、启停、备份、恢复、迁移、平移
- 创建 skill 的辅助流程

### 2.2 默认非目标

- 团队协作
- 云端账号体系
- 审批、权限、组织治理
- 通用 prompt / MCP / plugin 平台化扩张
- 以 marketplace 运营为核心的产品路线

## 3. UX 与产品表达规则

### 3.1 表达要求

- 默认把 `skill` 作为一等对象描述
- 避免把产品重新表述成“后台元 Agent”或“自动优化器”，除非是在描述 V1 历史
- 避免只强调“下载 skill”，要强调“看清、管住、优化 skill”

### 3.2 体验要求

- 可视化不是附属层，而是主价值
- 用户应在短时间内看懂自己的 skill 全貌
- 所有高风险操作都必须支持预览、备份、回滚
- 不要求用户理解底层目录结构才能完成主要操作

### 3.3 数据与结论要求

- 关于“这个 skill 有价值”或“这个 skill 应该演进”的结论，必须尽量给出证据
- 指标要优先可解释，不要制造虚假精度
- 对不完整数据明确标注 `partial` 或等价状态

## 4. 工程与仓库规则

### 4.1 代码管理

- 禁止未经允许新开分支；如确有必要，先向用户确认
- 遵守最小化提交原则：每个小主题改动积极提交
- 原则上仓库最终不应保留未提交改动
- 禁止不可恢复删除；只允许安全移动、备份或可回滚方式
- 单个代码文件原则上不超过 500 行；如确有必要，需说明理由并获得确认

### 4.2 实现原则

- 新增功能或 bug 修复后，优先补日志与可观测性建设
- bug 修复后，先自检确认无法复现，再认为问题已解决
- 测试尽量保持幂等或接近幂等
- 遇到启动、登录、环境配置、日志获取、打包、上传等容易出错的流程，要把经验沉淀到文档中，避免重复试错

### 4.3 文档更新原则

- 涉及产品定位、模块边界、信息架构变化时，应同步更新相应文档
- 如果文档描述仍停留在 V1 的“Skill Evolution Agent”定位，需要显式注明历史背景，避免与 V2 方向混淆
- 重要设计应优先写入 `docs/PRD.md` 或对应设计文档，而不是只停留在提交说明里

## 5. 决策优先级

发生冲突时，默认按以下顺序决策:

1. 用户明确要求
2. 安全与可回滚
3. V2.0 产品方向一致性
4. 可观测性与可解释性
5. 实现复杂度

如果某项实现明显偏离 “本地优先、skills 一等公民、围绕生命周期与演进” 这条主线，应先提醒，再执行。
