# LLM Skill 分析测试报告 V2（优化后）

**测试时间**: 2026-03-23  
**测试目标**: 验证优化后的 Prompt 能否让 LLM 给出更具体的 Skill 文档修改方案  
**测试模型**: DeepSeek Chat  
**测试结果**: ✅ 通过（10/10 分）- 输出质量显著提升

---

## 1. 测试对比

### 1.1 两次测试对比

| 维度 | V1（基础 Prompt） | V2（优化 Prompt） | 提升 |
|------|------------------|------------------|------|
| **Prompt 长度** | 约 9KB | 约 11KB | +约束说明 |
| **建议粒度** | 抽象建议（"增强查询能力"） | 具体修改（"在 ## 使用场景 添加..."） | ✅ 显著提升 |
| **Skill 范围约束** | ❌ 无约束（建议写脚本、改代码） | ✅ 明确约束（只能改 Skill 文档） | ✅ 关键改进 |
| **输出格式** | 简单建议列表 | 完整 Skill 文档片段 | ✅ 可执行 |
| **修改位置** | 不明确 | 精确到章节 | ✅ 可落地 |

### 1.2 关键改进点

**V1 问题**:
- 建议超出 Skill 范围（如"创建一个 audit_app_store_metadata 复合技能或脚本"）
- 建议过于抽象，无法直接执行
- 没有说明具体要修改 Skill 的哪个部分

**V2 改进**:
- 明确约束：只能修改 Skill 文档，不能写脚本、改代码
- 提供具体修改位置：在 ## 使用场景 章节添加...
- 提供完整 Skill 文档片段：可直接复制使用
- 说明预期效果：修改后 Agent 会如何表现

## 2. 优化后的 Prompt

### 2.1 新增约束部分

```markdown
## 重要约束（必须遵守）

1. **你的能力范围仅限于修改 Skill 文档**
   - Skill 是一个 Markdown 文档，包含指令和示例
   - 你可以编辑 Skill 的内容、结构、示例、约束条件
   - **你绝对不能**：修改代码、写脚本、修改配置文件、创建工具

2. **所有建议必须落实到 Skill 文档的具体修改**
   - 要说明：在 Skill 的哪个部分添加/修改什么内容
   - 要说明：修改后的 Skill 应该包含什么指令或示例
   - 不要给出抽象的"应该怎么做"，要给出"Skill 应该写什么"

3. **Skill 文档的典型结构**
   [Skill 是一个 Markdown 文档，包含以下章节]
   - # Skill 名称
   - ## 描述
   - ## 使用场景
   - ## 执行步骤
   - ## 约束条件
   - ## 示例
```

### 2.2 修改后的输出格式要求

```markdown
### Skill 文档修改方案

#### Skill: [skill-name]

**当前问题**:
[描述当前 skill 文档存在的问题]

**修改建议**:
1. 在 [章节] 添加：[具体内容]
2. 修改 [章节]：[具体内容]
3. 添加示例：[具体示例]

**修改后的 Skill 文档片段**:
```markdown
[提供修改后的 skill 文档内容，包含新增或修改的部分]
```

**预期效果**:
[说明这样修改后，下次遇到类似场景时，Agent 会如何表现]
```

## 3. LLM 输出对比

### 3.1 V1 输出（基础 Prompt）

**建议示例**:
> 1. 增强 `web_search_call` Skill 的"精准查询"能力
>    - 建议：为 Agent 内置或训练一个针对常见开发平台的"查询优化"子技能
>    - 具体实施：构建一个常见审核条款与官方文档章节的映射表

**问题**: 
- "构建映射表"超出了 Skill 文档的范围
- 没有说明要在 Skill 的哪个部分添加什么内容

### 3.2 V2 输出（优化 Prompt）

**建议示例**:
> **修改建议**:
> 1. 在 **## 使用场景** 章节添加：`App Store 审核问题诊断与回复`
> 2. 新增一个章节 **## App Store 审核合规性检查**，包含详细的执行步骤、约束条件和示例
> 3. 在 **## 示例** 章节添加一个针对审核拒信分析的完整示例
>
> **修改后的 Skill 文档片段**:
> ```markdown
> ## 使用场景
> - 开发与调试 Flutter/iOS 应用功能
> - **App Store 审核问题诊断与回复**
> 
> ## App Store 审核合规性检查
> 
> ### 核心原则
> - **先分析，后修改**：...
> - **内外核对**：...
> 
> ### 执行步骤（针对审核拒信分析）
> 1. **解析拒信**：...
> 2. **仓库现状分析**：...
> ```

**改进**:
- ✅ 明确指出修改位置（## 使用场景、## App Store 审核合规性检查）
- ✅ 提供可直接使用的 Skill 文档片段
- ✅ 完全在 Skill 文档范围内，不涉及代码修改

## 4. V2 详细输出

### 4.1 识别的 Skills

**Skill 名称**: `ios-dev` (位于 `./skills/ios-dev/SKILL.md`)

**关键洞察**:
> "从 Trace 看，Agent 没有显式调用一个名为 `ios-dev` 的 Skill 来执行结构化分析。相反，它通过一系列 `exec_command` 和 `web_search_call` 手动执行了本应由一个更完善的 'App Store 审核分析' Skill 来指导的步骤。"

这说明 LLM 能够理解：**当前缺少一个专门的 Skill 来指导这类任务**。

### 4.2 问题诊断

**精准诊断**:
> "当前 `ios-dev` Skill 可能更侧重于代码构建、测试、部署等开发任务，**缺乏针对 App Store 合规性、审核拒信分析、元数据检查等'上架后'或'审核阶段'特定场景的指导**。"

### 4.3 Skill 文档修改方案

#### 修改位置 1: ## 使用场景
**当前**: 只包含开发场景  
**修改**: 添加 `App Store 审核问题诊断与回复`

#### 修改位置 2: 新增 ## App Store 审核合规性检查
包含：
- **核心原则**（3条）
- **执行步骤**（5步详细流程）
- **约束条件**（3条）

#### 修改位置 3: ## 示例
**新增**: 完整的审核拒信分析示例，包含：
- 用户请求
- 拒信摘要
- Agent 应执行的操作（每步都有具体命令）
- 差距分析
- 输出方案

### 4.4 预期效果

> 1. 当下次遇到类似"分析审核拒信"的请求时，Agent 会优先调用此 Skill 中定义的标准流程
> 2. Agent 将更有条理地执行分析：解析拒信 -> 代码扫描 -> 配置检查 -> 链接验证 -> 生成差距报告
> 3. 避免 Agent 进行漫无目的的全局搜索，而是进行有针对性的检查

## 5. 关键发现

### 5.1 Prompt 工程的重要性

| Prompt 设计 | 输出质量 |
|------------|----------|
| 无约束 | 建议超出范围（写脚本、改代码） |
| 有明确约束 | 建议严格限制在 Skill 文档范围内 |

**关键约束**:
1. 明确能力边界（只能改 Skill 文档）
2. 要求具体修改位置（章节级别）
3. 要求提供可直接使用的文档片段

### 5.2 LLM 的理解能力

LLM 能够：
- ✅ 从 trace 中推断出缺少什么 Skill
- ✅ 理解 Skill 文档的结构和作用
- ✅ 给出符合约束的具体修改方案
- ✅ 提供可直接使用的文档内容

### 5.3 可执行性

**V1 建议**: 需要人工理解后再转化为 Skill 文档  
**V2 建议**: 可直接复制粘贴到 Skill 文档中使用

## 6. 架构验证

### 6.1 验证结论

优化后的测试进一步验证了架构设计的可行性：

```
原始 Trace + 约束 Prompt → LLM Agent → 可执行的 Skill 修改方案
      ↑                        ↑                ↑
   100%保留                理解约束          具体到章节
```

### 6.2 关键成功因素

1. **清晰的约束**: LLM 需要明确知道能做什么、不能做什么
2. **结构化输出**: 要求按固定格式输出，确保关键信息不遗漏
3. **具体化要求**: 要求提供可直接使用的文档片段，避免抽象建议
